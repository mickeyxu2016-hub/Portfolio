AI-Powered Mining Analytics

The Business Problem

At Komatsu, operations leads in the field can't answer their own data questions. When an operator wants to know how a machine is performing, the request travels through a three-step chain: operator → engineer → business analyst. The BA writes the SQL, runs it, formats a report, and sends it back.

That loop takes hours, sometimes days — by the time the answer arrives, the operational decision has already been made without it.

The core question: Can we replace that three-step chain with a single chat box?

Data Overview

We worked with industrial datasets generated by Komatsu electric shovels across three categories:

Equipment Health
Error episodes, event codes, and shift-level damage and abuse metrics tracking machine condition over time.
Operations Productivity
Shift reports covering dig hours, idle hours, payload amounts, and 10-minute utilization averages.
Dig Cycle Performance
Detailed timing, payload measurements, and machine positional metrics captured during active operation.

How It Works

The system pairs a custom Snowflake semantic model with Snowflake Cortex Analyst (LLM-backed text-to-SQL), wrapped in a Komatsu-branded Streamlit front-end deployed inside Snowflake.

01

Semantic Modeling

Translated business questions into a YAML semantic model — defined entities (machines, lineups, sites), relationships, verified queries, and a synonym layer so the model understands that "productivity," "tonnes per hour," and "throughput" all mean the same thing.

02

Verified Training Questions

Designed and validated seven training questions with Komatsu stakeholders, covering aggregations, exclusions, comparisons, and time-window analysis. Final accuracy benchmarked at 100%.

03

Streamlit Front-End

Built a chat interface with a one-click "Description of the data" button, a collapsible sample-question drawer, and a transparent SQL reveal below every answer. All styled with custom CSS to match Komatsu's brand.

04

Cortex Integration

Wired the front-end to Cortex Analyst via _snowflake.send_snow_api_request(), handling session state, retries, and rendering of returned SQL + tabular results.

05

Stakeholder Validation

Demoed the working app to Komatsu's BI lead and capstone faculty; iterated on UI, sample questions, and result formatting based on their feedback.

What Makes It Different

Most "AI chatbot" projects stop at "it generates SQL." Four design decisions made this one usable in production:

  • Pattern over phrasing. The semantic model encodes the concept of a product family, a metric, a time window. A user can ask "average productivity across one electric shovel family, excluding a specific unit" today, and tomorrow swap that unit for any other in the fleet, compare one product family against another, or exclude two units at once — and the chatbot still generates correct SQL. It understands the question pattern, not just the question.
  • Sanity checks on values, not just on syntax. Because the semantic model encodes the realistic range of each metric and the meaning of each unit, the chatbot refuses to silently return garbage. If a user types "how is the machine performing at 8,000,000°C," the system recognizes the value is impossible, flags it, and asks the user to confirm — suggesting plausible corrections like 80°C or 8°C. The model interprets the question instead of blindly executing it.
  • Discoverability built in. Because the semantic model also defines descriptions and synonyms, the same layer that powers SQL generation also answers "what's in this data?" — giving new users a zero-friction onboarding path with one click.
  • Trust through transparency. Every answer surfaces the generated SQL underneath. Non-technical users get the number; technical users get an auditable trail. This was the design choice the Komatsu BI lead flagged as the most important.

A Real Question, End to End

One of the demo questions, asked in plain English:

What is the average productivity across our electric shovel family, excluding the unit that's down for maintenance?

Three things have to happen for the system to answer this correctly:

  • Product hierarchy — know which machines belong to a given product family
  • Aggregation — compute an average over the productivity metric
  • Negation — translate the English word "excluding" into a SQL filter

The semantic model handles all three from one sentence. The chatbot returns the answer and the generated SQL side-by-side, so the user sees both what the number is and how it was computed.

Impact

  • Compressed a multi-day analyst loop into seconds for the seven validated question patterns
  • Freed business analysts from repetitive lookups to focus on higher-value analysis
  • Built a scalable architecture — new sensors or metrics can be added to the semantic model without rewriting the chatbot
  • Established a template Komatsu can extend across product families, additional sensor data, and external mining customers

The Business Problem

Credit, investment, and lending teams need to know one thing about a counterparty quickly: how likely are they to go bankrupt? The answer is usually buried in financial statements, ratio data, and market signals that arrive in different formats, on different schedules, from different sources.

Most of the time, the data engineering takes longer than the analysis. Analysts spend their week cleaning and joining files instead of generating insight — and the risk score that finally makes it to decision-makers is days old by the time it's ready.

The core question: Can we turn messy multi-source financial data into a single, fresh, decision-ready risk score?

S3 stores raw financial files from multiple sources. Glue handles schema inference, cleaning, and transformation. Redshift serves as the modeled, query-ready warehouse. SageMaker trains and serves the bankruptcy prediction model directly off Redshift features.

How I Built It

01

Data Ingestion (S3)

Landed raw financial statement and ratio data into versioned S3 buckets — partitioned by source and date so re-runs and audit trails are trivial.

02

Cleaning & Transformation (Glue)

Built AWS Glue jobs to standardize schemas, handle missing values, and engineer financial ratios (liquidity, leverage, profitability, efficiency) from the raw statement data — turning inconsistent inputs into a single clean feature table.

03

Modeled Warehouse (Redshift)

Loaded the clean features into Redshift, designed for analyst SQL access and model training. Built downstream views for ad-hoc risk analysis and dashboarding.

04

Model Training (SageMaker)

Trained a supervised classification model on labeled bankruptcy outcomes, pulling features directly from Redshift. Iterated on feature selection, class imbalance handling, and hyperparameter tuning to push validation accuracy to 92.52%.

05

Executive Recommendations

Translated model output into a one-page risk summary — flagging top-risk firms, the financial ratios driving the score, and recommended monitoring thresholds for credit / investment decisions.

What Made It Different

  • Pipeline-first, model-second. Most bankruptcy-prediction projects skip straight to the model on a clean Kaggle CSV. The harder, more realistic version is building the pipeline that produces the clean data — that's where most of the engineering value sits in production.
  • Auditable feature engineering. Every financial ratio was computed in Glue (not inside the notebook), so the inputs to the model could be traced back to the source rows and re-validated independently. No black-box features.
  • Analyst-friendly outputs. Risk scores landed back in Redshift alongside the source features so business users could query both the prediction and the reasoning in the same SQL call — no need to swap between systems.
  • Built to scale. Every stage in the pipeline is independently swappable. Add a new data source by adding a Glue job; swap the model by retraining inside SageMaker — the rest of the architecture doesn't move.

Impact

  • Delivered a 92.52% accuracy bankruptcy prediction model backed by a fully reproducible AWS data pipeline
  • Compressed multi-day, multi-file data prep into a repeatable pipeline run ready for analyst SQL access
  • Surfaced executive-level recommendations identifying the top financial-health red flags driving the model's predictions
  • Built a cloud-native architecture template that generalizes to any tabular risk-prediction use case — credit default, churn, fraud — by swapping the data source and label

What I Learned

  • Data engineering decides the model ceiling. The accuracy gains in the last few iterations didn't come from a better algorithm — they came from cleaner features and better imputation upstream in Glue. The pipeline is the model.
  • Reproducibility is a feature, not a checkbox. Versioned S3 buckets and idempotent Glue jobs sound boring until you have to re-run the whole pipeline to chase down a single anomaly — at which point they save the project.
  • Cloud is just plumbing — until it isn't. Decoupling storage (S3), compute (Glue), warehouse (Redshift), and ML (SageMaker) felt overkill for a class project, but it's exactly the pattern production teams use. Every architectural choice paid off the moment something needed to change.

Technologies Used

AWS S3
Raw and intermediate data storage, partitioned for incremental loads and full reproducibility of every pipeline run.
AWS Glue
Serverless ETL service for schema standardization, missing-value handling, and financial-ratio feature engineering at scale.
Amazon Redshift
Cloud data warehouse hosting the modeled feature tables, plus risk scores written back from SageMaker for analyst-friendly SQL access.
Amazon SageMaker
Managed ML platform for model training and evaluation — including hyperparameter tuning and class-imbalance handling for the bankruptcy classifier.
Python & SQL
Glue job logic, model training scripts, exploratory analysis, and the SQL views feeding the executive risk summary.
Pandas & scikit-learn
Local exploratory analysis and baseline model comparisons used to validate SageMaker results before promoting the final model.
